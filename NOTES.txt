# Multi-Touch Bug Investigation

## Current State (2026-04-21)

Game is functional with single-touch controls working well. Implemented pointer ID tracking for seamless finger handoff, but **multi-touch is not working on iPhone**.

---

## Bug Report

**Issue:** Second finger not recognized during multi-touch

**Symptoms:**
- ✅ First finger works perfectly (player moves with touch)
- ❌ Second finger completely ignored (no movement response)
- ❌ Lifting first finger while second is down does NOT activate second finger
- ❌ Only works when ALL fingers lifted, then one finger touches again

**Expected Behavior:**
- Second finger should immediately take control when touched
- Lifting first finger should maintain control with second finger
- Seamless handoff between fingers

**Tested On:** iPhone (iOS Safari)

## Hypotheses for Multi-Touch Failure

### Hypothesis 1: iOS Touch Action Interference
**Theory:** iOS Safari might be preventing multi-touch events due to CSS/browser defaults

**Possible Causes:**
- `touch-action` CSS property set to `none` or `manipulation`
- `e.preventDefault()` needed on pointer events
- Phaser's input system blocking secondary touches

**Test Steps:**
1. Add `e.preventDefault()` in `handleTouchStart`
2. Check CSS for `.phaser-game` or canvas element
3. Try adding `touch-action: none` to canvas CSS explicitly

**Code to Try:**
```javascript

    event.preventDefault(); // ← Add this

    // ...
}
```

# Enemy Type Brainstorming Session (2026-04-21)

## Inspiration
User discovered unexpected physics interactions between bullets and enemies during testing. These emergent behaviors inspired desire to create reactive enemy types that respond dynamically to player actions.

---

## Core Design Philosophy
- **Mobile-first design**: All enemy types must work well on 360x640 resolution
- **Physics-based interactions**: Enemies should respond to bullets in interesting ways
- **Touch-based gameplay**: Visual feedback should be clear during direct touch controls
- **Progressive complexity**: Start with simple enemies, introduce complex ones as player advances

---

## Enemy Type Concepts

### 1. Bouncer Enemies 🏓
**Description:** Round, metallic/glass-like enemies that reflect bullets
- **Behavior:** Bullets bounce off at angle based on hit position
- **Effect:** Creates unpredictable bullet trajectories player may need to dodge
- **Visual:** Shiny surface, metallic sound on impact
- **Implementation:** Calculate deflection angle based on bullet entry point

**Design Notes:**
- Could be a mid-game enemy type
- Creates skill ceiling: player learns bullet deflection patterns
- Risk/Reward: Can turn player's bullets against them or clear enemy clusters

---

### 2. Splitter Enemies 🧬
**Description:** Larger enemy that divides into smaller ones when hit
- **Behavior:** On first hit, splits into 3-4 smaller enemies
- **Effect:** Increases action density temporarily
- **Visual:** Cracking animation, then separation
- **Implementation:** Despawn parent, spawn children at same position with scattered velocities

**Design Notes:**
- Great for testing player's bullet management skills
- Smaller enemies could be faster/weaker or slower/tankier
- Could chain: smaller ones might split again on second hit

---

### 3. Reflector Enemies 🪞
**Description:** Flat enemy with mirror surface
- **Behavior:** Reflects bullets back toward player
- **Effect:** Dangerous if player gets too close
- **Visual:** Mirror-like, angled surface
- **Implementation:** Reverse bullet velocity vector

**Design Notes:**
- Creates tense "don't get too close" gameplay
- Could have directionality (reflects only from certain angles)
- High-skill ceiling: player must angle shots carefully

---

### 4. Pushable Enemies 🛷
**Description:** Enemies that build momentum when pushed by bullets
- **Behavior:** Gain velocity from bullet impacts
- **Effect:** Can push into other enemies for chain destruction
- **Visual:** Motion blur, particle trail
- **Implementation:** Add velocity vectors from bullet impacts, cap at max speed

**Design Notes:**
- Creates satisfying physics moments
- Heavy enemies = less pushed, light enemies = fly far
- Could enable "bullet billiards" style high-score plays

---

### 5. Absorber Enemies ⚡
**Description:** Enemy that collects bullets and charges up
- **Behavior:** Each bullet hit increases charge level
- **Effect:** Releases large counter-shot when full (or explodes)
- **Visual:** Glowing, growing aura, filling bar
- **Implementation:** Track hit count, trigger release at threshold

**Design Notes:**
- Creates risk assessment: shoot or avoid?
- Could have visual warning when nearly full
- Could vary: some explode, some release projectile, some release particles

---

## Mobile UX Considerations

| Enemy Type | Mobile Consideration | Risk Level |
|------------|---------------------|------------|
| **Bouncer** | Deflected bullets must be visible on small screen | Medium |
| **Splitter** | Too many small enemies could clutter 360px width | High |
| **Reflector** | Player might not notice danger in time | High |
| **Pushable** | Physics must feel responsive with touch timing | Medium |
| **Absorber** | Visual feedback must be clear without UI elements | Medium |

## Player Feedback Loop

**What makes these enemies exciting:**
1. **Unpredictability** - Player can't memorize perfect patterns
2. **Player agency** - Bullet placement matters, not just "hit vs miss"
3. **Emergent combos** - Bouncing off enemies, chain pushing, etc.
4. **Visual satisfaction** - Physics interactions feel impactful

**What could go wrong:**
1. **Too chaotic** - Screen becomes cluttered on small display
2. **Unfair deaths** - Player hit by own bullet without warning
3. **Learning curve** - New enemy types must feel intuitive
4. **Performance** - Many physics calculations per frame

## Technical Considerations

### Bullet Tracking
```javascript
// Example: Bouncer deflection logic
handleBulletHit(bullet, enemy) {
    // Calculate impact point relative to enemy center
    const impactX = bullet.x - enemy.x;
    const impactY = bullet.y - enemy.y;
    
    // Calculate deflection angle
    const angle = Math.atan2(impactY, impactX) + Math.PI; // +PI to reverse
    
    // Apply new velocity
    bullet.body.setVelocity(
        Math.cos(angle) * BULLET_SPEED,
        Math.sin(angle) * BULLET_SPEED
    );
    
    // Visual feedback
    this.tweens.add({
        targets: enemy,
        alpha: 1,
        duration: 100,
        yoyo: true,
        repeat: 1
    });
}
```

### Splitter Spawning
```javascript
// Example: Splitter logic
handleSplitterHit(enemy) {
    // Store split pattern
    const splitPattern = [
        { offsetX: -20, offsetY: -10, velocity: 50 },
        { offsetX: 20, offsetY: -10, velocity: 50 },
        { offsetX: 0, offsetY: 20, velocity: 75 },
        { offsetX: -15, offsetY: 15, velocity: 60 }
    ];
    
    // Spawn children
    splitPattern.forEach(pattern => {
        const child = this.add.rectangle(
            enemy.x + pattern.offsetX,
            enemy.y + pattern.offsetY,
            ENEMY_SIZE_SMALL,
            ENEMY_SIZE_SMALL,
            COLOR_ENEMY_CHILD
        );
        // ... physics setup, velocity, etc.
    });
    
    // Despawn parent with particle effect
    enemy.destroy();
    this.createParticleSystem(enemy.x, enemy.y);
}
```
